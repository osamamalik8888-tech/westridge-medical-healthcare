# Build Guide

A plain-language walkthrough of the two commands that turn this source
code into a running website: `npm install` and `npm run build`.

## Before you start

You need [Node.js](https://nodejs.org) version 20 or newer. Check what you
have:

```bash
node --version
```

If that prints something below `v20`, install a newer version from
nodejs.org first — everything below assumes 20+.

## Step 1: `npm install`

```bash
npm install
```

This reads `package.json`, downloads every package this project depends
on, and puts them in a `node_modules` folder (which is intentionally not
included in this handoff — it's regenerated by this command, and it's
large). Here's what's actually being installed and why:

| Package | What it's for |
|---|---|
| `next`, `react`, `react-dom` | The framework itself — routing, rendering, the dev server |
| `tailwindcss`, `@tailwindcss/postcss` | The styling system used for every visual style in the project |
| `motion` | Component-level animation (the hero entrance, header dropdown, mobile menu) |
| `gsap` | Scroll-triggered reveals (services list, doctor cards, the "why Westridge" grid) |
| `three` | The ambient particle background on the closing appointment section |
| `react-hook-form`, `zod`, `@hookform/resolvers` | Form handling and validation for the appointment and careers forms |
| `clsx`, `tailwind-merge` | Small utilities for combining CSS class names safely |
| `sharp` | Image optimization during the build |
| `vitest` | Runs the automated test suite (`lib/*.test.ts`) |
| `@vercel/analytics` | Privacy-friendly traffic analytics, active only when deployed on Vercel |
| `typescript` | Type-checking, so mismatched data shapes get caught before they ship |
| `eslint`, `eslint-config-next` | Code-quality linting |

This step needs an internet connection and usually takes one to a few
minutes depending on your connection speed. You'll see a `node_modules`
folder appear, and a `package-lock.json` file get created or updated —
commit that lock file to Git, since it pins the exact versions everyone
on the project (including CI) should use.

**If it fails:** the error almost always names a specific package. Search
that package name plus the error text — it's rarely something wrong with
this codebase specifically, usually a Node version mismatch or a network
hiccup. Re-running `npm install` after a failed attempt is safe.

## Step 2: `npm run build`

```bash
npm run build
```

This is the real compile check. It runs, in order:

1. **TypeScript type-checking** — every file is checked for type
   mismatches (e.g. passing a number where a string is expected). This is
   the step this project could *not* run inside the sandbox it was built
   in (no internet access there to install the type definitions), so this
   is genuinely the first time the full codebase gets checked this way.
2. **Next.js compilation** — every page and component gets compiled and
   bundled, code-split by route so each page only ships the JavaScript it
   actually needs.
3. **Static generation** — pages that don't need per-request data (which
   is most of this site — the blog, services, about, contact, etc.) get
   pre-rendered to plain HTML at build time, so they load instantly and
   don't need a server to compute them on every visit.
4. **Image optimization setup** — `next/image` usages get wired up for
   on-demand resizing and format conversion (via `sharp`).

**A clean build ends with a route summary** — a table listing every page,
its size, and whether it's static or dynamic. No red text, no `Error`,
exit code 0.

**If it fails:** the error output tells you the file and line. Given how
much of this project was verified by hand (bracket balance, import
resolution, Tailwind class validity, and strict TypeScript checks on every
file that could be checked without network access) rather than by an
actual compiler, a failure here — if there is one — is the single most
useful piece of information you could send back for a fix, since it's the
one category of bug that couldn't be caught until now.

## After a successful build

```bash
npm run start
```

Runs the production build locally at `http://localhost:3000`, so you can
click through the real, optimized site before deploying — different from
`npm run dev`, which runs an unoptimized development version meant for
active coding, not for judging final performance.

## Running the test suite

```bash
npm run test:run
```

Runs the real unit test suite (`lib/*.test.ts`) — the pure business-logic
functions (health calculators, symptom-checker triage, the appointment/
careers form validation, and the time-formatting helper that had a real
bug caught and fixed earlier in this project). These don't need a browser
or a database; they run in plain Node and finish in under a second.

```bash
npm run test
```

Same thing, but in watch mode — re-runs automatically as files change,
useful while actively editing anything in `lib/`.

**A note on scope:** these tests cover the pure logic layer on purpose, not
full component or page rendering. Vitest doesn't yet support testing async
Server Components (which `app/blog/[slug]/page.tsx` is), so component-level
testing would need Playwright or a similar browser-based tool — a
reasonable next investment, not a gap in what's here.

## Optional: lint on its own

```bash
npm run lint
```

Runs just the code-quality checks without a full build — faster feedback
while making further changes. This is also what runs automatically in CI
(`.github/workflows/ci.yml`) on every push.
